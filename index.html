Overview

ESP.App: Internal Operations Control Plane

ESP.App is a browser-based internal platform for turning manual energy operations workflows into repeatable application workflows. It provides user-facing screens for reviewing operational data, running calculations, generating reports, managing schedules, tracking workflow execution, and administering access.

Product

Internal operations control plane

Domain

Energy operations and demand response

Role

Product-minded data scientist / analytics engineer

Stack

React, TypeScript, FastAPI, Python, SQLAlchemy, Celery, Redis, PostgreSQL, Azure SQL

The Problem

Critical workflows lived outside a product surface

Energy operations workflows often depend on spreadsheets, scripts, analyst memory, file drops, and manual handoffs. That makes recurring work hard to review, hard to schedule, hard to audit, and hard to scale across programs. ESP.App was built to bring those workflows into a single internal product surface with clearer execution, review, and operational visibility.

  • Manual spreadsheet-driven processes
  • Recurring calculations and reporting workflows
  • Multiple datasets and program-specific rules
  • Limited visibility into what ran, when it ran, and what was produced
  • Need for reusable workflows that analysts and operators could actually use

What I Built

A control plane over reusable domain logic

ESP.App separates reusable business logic from the browser-based product surface. The result is a maintainable two-layer model: a shared Python domain library and an internal web application that exposes those capabilities through workflows, review screens, schedules, logs, downloads, access controls, feedback tools, and admin interfaces.

ESP.Artifacts

Python domain layer

A shared Python domain library containing reusable business logic for database access, external data clients, baseline and settlement calculations, data quality checks, payment file generation, statement generation, performance report generation, and program-specific automation packages.

ESP.App

Internal web application

A React/FastAPI internal web application that exposes that domain logic through user workflows, review screens, schedules, run logs, downloads, access controls, feedback tools, and admin interfaces.

Product Workflow

From operational data to repeatable workflow

ESP.App gives users a structured surface to filter, review, validate, calculate, and export operational results. Instead of running one-off scripts or editing spreadsheets, users work through repeatable application screens connected to backend services and reusable Python modules.

  1. User selects operational scope.
  2. App loads relevant events, assets, or workflow rows.
  3. User reviews, filters, validates, or triggers an action.
  4. Backend services call reusable artifact modules.
  5. Outputs are written, logged, exported, or made available for downstream review.

Execution And Observability

Scheduled execution with run visibility

The app supports scheduled workflows, ordered workflow steps, run status tracking, logs, and execution history. This turns recurring operational work into observable workflows rather than hidden scripts.

  • Scheduled jobs
  • Workflow steps
  • Celery dispatch
  • Run logs
  • Success/failure visibility
  • Operational traceability
  • Reduced dependence on manual triggers

Artifact Generation

Generating business-facing outputs from operational workflows

ESP.App does not stop at internal tables. It supports downstream artifact generation: reports, exports, downloadable files, and structured outputs used by operational stakeholders. It helps turn reviewed operational data into structured outputs and creates a repeatable path from data review to report generation.

Documentation

Documentation inside the workflow

ESP.App includes embedded page-level and program-level guidance so users can access instructions without leaving the operational workflow. This helps reduce dependence on tribal knowledge and makes program-specific processes easier to follow.

Admin And Feedback

Admin tooling for a real internal product

The platform includes authentication, user roles, page access patterns, activity tracking, admin views, and page feedback. These features matter because internal tools still need operational ownership, usage visibility, and feedback loops.
